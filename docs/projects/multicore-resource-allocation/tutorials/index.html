Tutorials: Multicore Resource Allocation

Practical guides for setting up platforms, profiling task phases, running allocation frameworks, and interpreting traces on PREEMPT-RT and LITMUSRT.

1. Environment setup

Install PREEMPT-RT or LITMUSRT with support for cache partitioning (e.g., CAT) and memory bandwidth control (e.g., MBM/MBA). Ensure perf and tracing are available for phase profiling.

  • Kernel: enable real-time patch, perf events, and resource-control features (CAT/MBM).
  • Tooling: build schedulers/runtimes (DNA/DADNA, Omni, DECNTR, RASCO) and tracing scripts.
  • Baseline: capture hardware topology (cores, LLC ways, memory controllers) for experiments.

2. Profiling task phases (DNA/DADNA)

Profile representative runs to discover phase boundaries and resource sensitivity before enabling dynamic allocation.

  • Collect per-phase metrics (cycles, LLC misses, bandwidth) with perf or hw counters.
  • Identify phases with distinct resource needs and map them to cache/bandwidth profiles.
  • Train the runtime to detect phase transitions and update allocations accordingly.

3. Running dynamic allocation

Enable the runtime scheduler, register tasks, and allow it to reallocate cache ways and bandwidth based on observed phases or deadlines.

  • Configure per-task priorities/deadlines and initial cache/bandwidth budgets.
  • Start allocation daemon; verify reallocation events via logs and trace markers.
  • Measure latency distributions and deadline miss ratios before/after dynamic allocation.

4. Multi-mode transitions (Omni)

Use Omni to plan allocations per mode and keep transitions schedulable.

  • Define modes, task sets, and transition guards/triggers.
  • Compute feasible allocations per mode and during transitions to avoid contention spikes.
  • Trace mode switches and verify deadlines are maintained throughout transitions.

5. Control co-design (DECNTR)

Pair controller variants with resource budgets so closed-loop systems stay in safe sets.

  • List controller options (sampling periods, computational cost) and safety envelopes.
  • Choose allocations and controller variants jointly; deploy and monitor state trajectories.
  • Validate robustness to disturbances while meeting deadlines.

6. DAG workloads (RASCO)

Co-design scheduling and per-phase budgets for dependent tasks.

  • Model the DAG with execution times per resource configuration.
  • Compute placements and budgets that satisfy end-to-end deadlines.
  • Instrument traces to confirm critical-path timing under dynamic allocation.

7. Evaluation checklist

  • Latency: report median, 95th/99th percentile, and deadline-miss rate.
  • Overhead: quantify allocation/scheduling overhead and cache/bandwidth reprogramming costs.
  • Ablations: compare static partitions vs. dynamic allocation per framework.
  • Reproducibility: document kernel config, hardware topology, workloads, and seeds.
